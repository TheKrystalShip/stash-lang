# RFC: Hermetic VM — engine and async-child isolation

> **Status:** Draft
> **Owner:** Cristian Moraru
> **Created:** 2026-06-02
> **Slug:** hermetic-vm
> **Milestone:** embedding

## Summary

Make a Stash `VirtualMachine` instance **hermetic** in two distinct senses that the design note (`Hermetic VM — Isolation Design (Embedding Phase 2).md`) keeps strictly separate:

- **Engine ↔ engine** — two `VirtualMachine` instances in the same .NET process must share *nothing observable* through stdlib. Today, `env.chdir` / `env.set` / `env.unset` / `process.spawn` reach past the VM into `System.Environment`, signal handlers register in a global `ConcurrentDictionary` keyed by signal name (last-writer-wins on SIGTERM), and the shared `Chunk.ICSlots` array gets concurrently written from each VM's IC path — a *Critical* silent-corruption hazard that is already live under async (parent + child execute the same chunk).
- **Parent ↔ async-child (within one engine)** — `async fn` already forks a child VM on a thread-pool thread (`VirtualMachine.Async.cs:64-90`) and shares reference-typed globals with the parent **by reference** (`new Dictionary<string, StashValue>(_globals)` shallow-copies the dict; every `StashArray`/`StashDictionary`/`StashInstance` global stays shared). `_importStack` is shared by reference across threads. The runtime collection types are not thread-safe → concurrent mutation can corrupt internal storage today.

The fix decided up-front (design note §2): for each async-child global entry, if the value is **frozen** (`readonly`/`freeze`, the phase-1 `IsFrozen` predicate) share by reference; otherwise **deep-clone** (cycle-safe, reusing phase-1's `DeepFreeze` walker shape, with a `visited` set that throws on a cycle for v1). For process-global state, virtualize cwd / env / signals onto per-VM state on `VMContext`, and route **all** stdlib path/env/spawn/signal sinks through that per-VM view.

Multi-engine isolation is the deliverable — it is *tested*, not asserted: this spec carries the acceptance suite from design note §8 (two-engine cwd, two-engine env, async-race stress, IC-slot correctness under concurrent load, SIGTERM multiplex) as its own phase that goes green.

## Motivation

Three different audiences feel the pain today, in increasing order of impact:

1. **Live async correctness (everyone running async Stash, no embedding involved).** `VirtualMachine.Async.cs:50` does `new Dictionary<string, StashValue>(_globals)` — the dict is new, but `StashValue` is a struct carrying an `_obj` reference, so every reference-typed global is shared across the fork. A non-frozen `StashDictionary` mutated by parent and async-child simultaneously races on its underlying storage — concurrent writers to the same `Dictionary<TKey,TValue>` can corrupt it (this is .NET-documented behavior, not a Stash claim). Parents who mutate after spawn see the child's writes; children who mutate see the parent's. Either is a real bug; today it is "don't do that" instead of "the runtime won't let it corrupt."
2. **Silent IC-slot corruption under concurrent load (everyone running async or two engines, no embedding involved).** `Chunk.ICSlots` is a mutable `ICSlot[]` (`Chunk.cs:63`) attached to a *shared* compiled `Chunk`. Each VM executing that chunk writes into the same array (`VirtualMachine.Collections.cs:266,296,287` writes `ic.State` and `ic.Guard`; `VirtualMachine.Functions.cs:587,620`). Cross-VM writers can corrupt cached field offsets — wrong-field reads, no diagnostic. This is the §4 **Critical** row, live today the moment any program forks an async child.
3. **The embedding initiative is blocked on this.** Phase 3 (`Stash.Hosting`) sells "a `StashEngine` is a `lua_State`." That promise is observably untrue today: engine A's `env.chdir("/tmp")` changes the .NET process cwd, so engine B's `fs.realpath(".")` returns `/tmp` too. Two engines registering SIGTERM cleanup with `signal.on` overwrite each other in `SignalImpl.SignalHandlers` — only one runs on the signal, the other's locks leak. We can't ship a host SDK whose first integration test fails on these.

Cost of doing nothing: ship phase 3 with a Lua-state pitch that fails the first multi-engine test the host writes, plus keep the live correctness bugs (1) and (2) in the language regardless.

## Goals

- Two `VirtualMachine` instances in one process share **no observable stdlib state**: independent cwd, independent env overlay, independent signal handler list (with SIGTERM dispatching to *every* registered handler), no `Console` fallthrough when `EmbeddedMode` is on, and independent IC-slot writes (no cross-engine corruption).
- An async-child VM, when forked from a parent, gets:
  - frozen globals shared by reference (immutable can't race), and
  - non-frozen reference-typed globals **deep-cloned** at fork (cycle-safe via a `visited` set; throw on cycle for v1).
  - Its own `_importStack` snapshot (per design note §2; we are precise about *which* fork sites snapshot — see Implementation Path point 5 below — module-load same-thread sharing of `_importStack` is *deliberate* circular-import detection and must NOT be isolated).
- `process.spawn`/`process.exec`/`process.pipeline`/`process.daemonize` and any other `ProcessStartInfo` site inherit the **VM's** view of cwd and env (`ctx.WorkingDirectory`, `ctx.EnvVars` merged with real process env per documented merge semantics), not the host process's view.
- An **isolation acceptance suite** (design note §8) is green and is part of `final_verify`, so any regression in any future feature that re-introduces an engine leak fails CI.
- Engine construction stays cheap (we share the immutable `Chunk` graph + stdlib registries; per-VM allocation is only mutable IC-slot clones + per-VM cwd/env state). Numeric target is not asserted here — phase 3 owns "is a pool needed?".

## Non-Goals

- **Building async.** Async is fully built (`StashFuture`, `Task<object?>`, `await`); this spec isolates the existing threaded forks, it does not introduce parallelism.
- **A VM pool.** Demoted by the milestone charter to a phase-3 concern; not specced here.
- **The `Stash.Hosting` host SDK / typed marshalling / `InvokeAsync`.** Phase 3.
- **The freeze primitive.** Phase 1 (`readonly` modifier) shipped it; this spec consumes `IsFrozen` and `DeepFreeze` (`Stash.Core/Runtime/RuntimeValues.cs:318`, `Stash.Core/Runtime/Types/{StashArray,StashDictionary,StashInstance,StashTypedArray,StashNamespace}.cs`).
- **Cycle-preserving clone.** v1 = detect-and-fail on cycle (visited set; throw a typed error explaining how to break the cycle, typically by `freeze`-ing one node). Cycle-preserving clone is a future enhancement.
- **A new "embedded profile" stdlib subset.** Per design note §5, the full stdlib stays available and hosts opt out via `StashCapabilities`.
- **Migrating `Console.Out`/`Error`/`In` defaults for CLI-mode.** CLI mode keeps its current `Console.*` defaults; the change is gating that fallthrough on `EmbeddedMode == false` so embedded hosts get an explicit no-Console policy.
- **`HttpClient`-per-engine.** Low severity; leave the static singleton, add a follow-up backlog stub if a host raises it.
- **Per-engine `Random.Shared` reseed.** Low; deferred.
- **REPL prompt / process delegate statics.** High for CLI test isolation, low for embedded; deferred unless cheap as part of the multiplexed signal work.

## Design

### Surface

User-visible Stash surface changes (small — most of this is plumbing):

- **`env.set` / `env.unset` / `env.get` / `env.has` / `env.all` / `env.withPrefix` / `env.loadFile` / `env.saveFile`** now read/write a **per-VM** environment **overlay**, not the real `System.Environment`. Reads consult the overlay first, then fall back to the real process env so that variables set outside the engine (typical embedded use) remain visible. Writes never touch `System.Environment`. `env.all()` returns the merged view (overlay + process env, overlay wins).
- **`env.cwd` / `env.chdir` / `env.popDir` / `env.dirStack` / `env.withDir`** now read/write a **per-VM** `WorkingDirectory`. `env.chdir` does *not* mutate `System.Environment.CurrentDirectory`. Tilde expansion stays at the boundary, unchanged.
- **All path-resolving stdlib sinks** (anywhere `Path.GetFullPath(relativeArg)` or `Directory.GetCurrentDirectory()` was used to resolve a user-supplied path) route through the per-VM cwd. This is invisible at the surface for users running one engine, but becomes the load-bearing engine-isolation invariant in embedded scenarios.
- **`process.spawn` / `process.exec` / `process.pipeline` / `process.daemonize` / `process.replace`** inherit the VM's cwd and env by default (`ProcessStartInfo.WorkingDirectory = ctx.WorkingDirectory`; `psi.Environment` seeded from the merged overlay+real env). Per-call `ExecOptions { cwd, env }` continues to override (existing behavior, unchanged).
- **`signal.on(SIGTERM, fn)`** in two engines no longer overwrites — both register, and on SIGTERM both run (multiplexed handler). `signal.off(SIGTERM)` removes *this VM's* handler only (identity via `IInterpreterContext`).
- **Async semantics change (with migration note).** Mutating a non-frozen captured global *from inside an async body* used to (racily) mutate the parent's value through the shared reference. It now mutates a **deep-clone** local to the child — `call-local`, invisible to the parent. The recommended pattern is to communicate via return values (`await`, `task.all`). If the host *wants* shared-by-reference semantics for cross-async-task communication, the value must be `freeze`d (it then shares; writes throw — a deliberately immutable hand-off). This is a behavior change; the migration note in the spec section calls it out.
- **Cycle in a deep-clone target throws** `ValueError` (or a more specific `CycleError` if we add one — see Open Question) at fork time, with the path to the cycle in the message. Recommendation: `freeze` one node on the cycle.

### Semantics

**Per-VM virtual cwd**. `VMContext.WorkingDirectory` is initialized from `System.Environment.CurrentDirectory` at VM construction (only at construction — never re-read). Every stdlib sink that previously called `System.Environment.CurrentDirectory` or `Directory.GetCurrentDirectory()` calls `ctx.WorkingDirectory` instead, and every sink that called `Path.GetFullPath(relativeArg)` calls a single `ctx.ResolveAgainstCwd(relativeArg)` helper that combines `ctx.WorkingDirectory` with the user path before normalizing. The `DirStack` already lives on `VMContext`; only the *side effect* changes (no longer touches the real cwd). `env.withDir` becomes a try/finally on `ctx.WorkingDirectory` instead of the real cwd.

**Per-VM virtual env overlay**. `VMContext.EnvVars` is a `Dictionary<string, string?>` (null marks "explicitly unset", to shadow a real process env var). It starts empty. Reads (`env.get`, `env.has`, `env.all`, `env.withPrefix`, anywhere the stdlib reads an env var): first consult overlay (a present null = "unset, return null/false/skip"; a present non-null = that value; absent = fall back to `Environment.GetEnvironmentVariable(name)`). Writes (`env.set`, `env.unset`, `env.loadFile`): write to overlay only. `env.saveFile` writes the merged view (overlay over real).

**Non-overlaid env reads (identity vs mutable state).** A handful of reads are about *process identity*, not the mutable env state, and stay raw: `env.home` (user profile path), `env.hostname` (machine name), `env.user` (login name). These are not "environment variables" in the overlay-able sense — they're constants for the process lifetime. The Cross-Cutting Concerns guard's exemption list documents this explicitly so a future migrator does not mistakenly route them through the overlay.

**`process.spawn` inheritance**. The internal `ApplyCwdAndEnv` helper (`Stash.Stdlib/BuiltIns/ProcessBuiltIns.cs:1120`) is the single funnel; it now seeds `psi.WorkingDirectory` from `ctx.WorkingDirectory` when caller did not specify `cwd`, and seeds `psi.Environment` from the merged env overlay + real env (overlay wins) when caller did not specify `env`. Per-call `cwd` / `env` overrides continue to take precedence (existing behavior). Every `new ProcessStartInfo` site (≈9 today, grepped in `ProcessBuiltIns.cs`) routes through this funnel — that funnelization is part of the cross-cutting omission guard.

**Multiplexed signal registry**. Replace `SignalImpl.SignalHandlers` (today `ConcurrentDictionary<string, (Context, Handler, Registration)>` — last-writer-wins) with `ConcurrentDictionary<string, List<(IInterpreterContext, IStashCallable, PosixSignalRegistration?)>>` under `SignalLock`. On signal fire, snapshot the list for that signal and invoke every entry in registration order, each wrapped in `try/catch` (errors in one handler must not prevent others). `signal.off(signalName)` removes entries whose `Context` reference equals the calling VM's `_context` (identity match). The single `PosixSignalRegistration` per signal is created on the first registration and disposed when the last entry for that signal is removed.

**No `Console.*` fallthrough in embedded mode**. `VirtualMachine.cs:165-172` currently has `Output`/`ErrorOutput`/`Input` default to `Console.Out`/`Console.Error`/`Console.In` via `VMContext` defaults. New rule: when `EmbeddedMode == true`, defaults become `TextWriter.Null` / `TextReader.Null` (host must wire explicit writers — `StashEngine` already does this at `StashEngine.cs:39-41`). CLI mode is unchanged.

**Per-VM IC-slot clone**. On the *first* execution of a `Chunk` by a `VirtualMachine` instance, the VM clones `chunk.ICSlots` (and recursively for every nested function `Chunk` reachable through `Constants` / closures) into a per-VM `Dictionary<Chunk, ICSlot[]>` (or equivalent) and the IC fast/slow paths in `VirtualMachine.Collections.cs:266,296` / `VirtualMachine.Functions.cs:587,620` index into the *VM's* array, not `chunk.ICSlots`. Allocation strategy and exact representation (lazy-on-first-execute vs eager-walk-at-Execute-entry; per-Chunk dict vs flat array keyed by chunk id) is an implementation choice the implementer owns within phase 2A; the *invariant* is "no two VMs ever write to the same IC array."

**Async-child freeze-or-clone**. In `SpawnAsyncFunction`, replace the shallow `new Dictionary<string, StashValue>(_globals)` with a call to a single `IsolationHelpers.BuildChildGlobals(parentGlobals)` that, for each entry:
- primitive (`!IsObj` or string) → copy by value (struct semantics or string immutability).
- `StashValue` whose `_obj` is **frozen** per `IsFrozen` (across `StashArray` / `StashDictionary` / `StashInstance` / `StashTypedArray` / `StashNamespace`) → share by reference.
- otherwise (mutable reference) → call `RuntimeValues.DeepClone(value, visited)` — a sibling to phase-1's `DeepFreeze` walker, reusing the cycle-safe `HashSet<object>(ReferenceEqualityComparer.Instance)` pattern at `Stash.Core/Runtime/RuntimeValues.cs:325`. v1: throw on cycle.

The same helper is also called from the **other cross-thread child-VM construction site** discovered during §4 re-verification (see Implementation Path point 5), each tagged with its sharing semantics.

### Implementation Path

The end-to-end path that must survive every phase:

1. **`VMContext` carries the per-VM mutable view of process state.** Add `WorkingDirectory` (init from `System.Environment.CurrentDirectory` at construction; never re-read) and `EnvVars` (overlay dict; null marks explicit-unset). The existing `DirStack` becomes the user-facing stack on top of `WorkingDirectory`.

2. **Stdlib path/env/spawn/signal sinks route through `VMContext`.** Every today-direct call to `System.Environment.CurrentDirectory` / `GetEnvironmentVariable` / `SetEnvironmentVariable` / `GetEnvironmentVariables` / `Directory.GetCurrentDirectory` / `Path.GetFullPath(relativeArg)` in stdlib code becomes a call through `IInterpreterContext` (new members: `WorkingDirectory`, `GetEnv`, `SetEnv`, `UnsetEnv`, `AllEnv`, `ResolveAgainstCwd`). Internal `ApplyCwdAndEnv` in `ProcessBuiltIns` is the single funnel that seeds `ProcessStartInfo` from the VM view. Construct (banned-API meta-test) prevents new direct accesses outside the exemption list — see Cross-Cutting Concerns.

3. **`SignalImpl` multiplexes.** Storage shape changes from "one handler per signal" to "list of (context, handler, registration) per signal." On fire: iterate the list, each handler `try/catch`'d. `signal.off` removes the calling VM's entries by `Context` identity. Last entry removal disposes the underlying `PosixSignalRegistration`.

4. **Per-VM IC-slot clone on Chunk-load.** A `VirtualMachine` lazily materializes a private `ICSlot[]` for every `Chunk` it executes — top-level passed to `Execute(chunk)` and every nested chunk reachable through `Constants` / `MakeClosure`. The `ICSlots`-indexed opcodes index the *VM's* per-chunk array, not `chunk.ICSlots`. `chunk.ICSlots` continues to exist as the *template* (`new ICSlot[]` with `ConstantIndex` pre-filled by `ChunkBuilder.cs:313-317`); the clone copies that template per VM. This closes the §4 *Critical* row.

5. **Single `BuildChildGlobals` helper at every cross-thread child-VM construction site.** §4 names `SpawnAsyncFunction` as the leaking fork, but grepping the codebase shows **five** `new VirtualMachine(...)` sites that build a child from parent state. Each is classified and treated explicitly:
   - `VirtualMachine.Async.cs:64-90` — `SpawnAsyncFunction`: **cross-thread**; must freeze-or-clone globals + snapshot `_importStack`.
   - `VirtualMachine.Modules.cs:105-112` — module-load child: **same-thread**; circular-import detection deliberately requires shared `_importStack` (`Modules.cs:77`). Must NOT isolate `_importStack`. Globals here are a copy used as the module-VM's *initial* env, then filtered through `Exports` — no cross-thread hazard. Document as "no isolation needed; same-thread, deliberately shares import stack."
   - `VMContext.cs:323-343` — `InvokeCallbackDirect` background-thread branch (when current thread ≠ `MainThreadId`): **cross-thread**; same shape as the async fork. Must use `BuildChildGlobals`.
   - `VMContext.cs:316-322` (sibling `if`) — `InvokeCallbackDirect` same-thread branch: stays inline (no new VM is allocated on this branch — it executes via `ActiveVM.ExecuteVMFunctionInlineDirect`); no fork; no isolation needed.
   - `VMTemplateEvaluator.cs:96` — template expression eval: **same-thread**, transient single-expression VM with `evalGlobals = scope.Flatten()` (already a fresh dict); no async hazard. Document as "no isolation needed."
   - `StashEngine.cs:203` — engine construction (the host's primary VM): **engine root**, not a child; not a fork. Out of scope.
   The cross-cutting omission guard for "child VM construction" makes `BuildChildGlobals` the single source of truth for cross-thread forks; new fork sites either route through it or appear on the meta-test's exemption list with a *justified* same-thread reason.

6. **`Console.*` fallthrough gated on `!EmbeddedMode`.** `VMContext` constructor (or `VirtualMachine.cs:165-172`'s property getters) returns `Null` writers/reader when `EmbeddedMode == true` and the host has not set explicit writers. `StashEngine.cs:39-41` already sets explicit ones, so CLI behavior is unchanged.

7. **Tests prove the deliverable.** The isolation acceptance suite from design note §8 is its own phase, running:
   - Two-engine cwd isolation (`engine A.env.chdir("/tmp"); engine B.fs.realpath(".")` unaffected).
   - Two-engine env isolation (`A.env.set("X","1"); B.env.get("X")` is null).
   - Async-race stress (N tasks mutating a non-frozen captured collection: no corruption + call-local; repeat with `freeze`'d collection: shared + writes throw).
   - **IC-slot correctness under concurrent load** with a **nested-chunk** field access (per advisor: a top-level-only test passes even when nested-chunk IC arrays still leak — the test must execute field access from inside a closure or method to have teeth).
   - SIGTERM multiplex (both engines' cleanups fire).

8. **Toolchain compatibility verified per `.claude/language-changes.md`** at the relevant phase. `env.*` semantics change is the visible-behavior change; the doc impact matrix is recorded in Cross-Cutting Concerns and updated in phase 2B.

## Acceptance Criteria

End-to-end behavior that proves the feature works:

1. **Two-engine cwd isolation**: a host constructs `var engineA = new StashEngine(); var engineB = new StashEngine();`, calls `engineA.Execute("env.chdir('/tmp')")`, then `engineB.Execute("fs.realpath('.')")` returns the *original* process cwd, not `/tmp`. The real `System.Environment.CurrentDirectory` is also unchanged.
2. **Two-engine env isolation**: engineA sets `env.set("X", "1")`; `engineB.Execute("env.get('X')")` returns `null`; `System.Environment.GetEnvironmentVariable("X")` returns null too.
3. **Async-race stress (call-local mutation)**: spawn 100 async tasks each mutating a captured non-frozen `StashDictionary`. After `task.all`, the *parent*'s dict is unchanged, every child's mutations were applied to its own clone, and there is no `Dictionary` corruption (no `InvalidOperationException` from concurrent modification, no swallowed write).
4. **Async-frozen sharing**: repeat (3) with the dict `freeze`d before spawn. Children share the dict by reference; their `dict.set(...)` writes throw `ReadOnlyError`; the parent's view is unchanged.
5. **IC-slot correctness under concurrent load (nested-chunk teeth)**: a chunk whose hot path reads `obj.field` from inside a **nested function** is executed concurrently across two engines on different `StashStruct` shapes. Neither engine ever sees a wrong-field read; field offsets in their IC slots are independent. (Top-level-only would pass even if nested chunks leaked — this test is engineered to fail without the recursive clone.)
6. **SIGTERM multiplex**: two engines each register `signal.on(SIGTERM, cleanup_fn)`; sending SIGTERM to the process invokes *both* `cleanup_fn`s. `signal.off(SIGTERM)` in engine A leaves engine B's handler registered (subsequent SIGTERM only runs B).
7. **`process.spawn` inherits the VM's view**: in engineA, `env.chdir("/tmp"); process.spawn("/bin/pwd")` reports `/tmp`; in engineB at the same time, `process.spawn("/bin/pwd")` reports the original cwd.
8. **`final_verify` is the full `dotnet test` suite (no namespace-exclusion filters), and it is green.** The full suite includes `NoProcessGlobalLeakMetaTests` and `ChildVMConstructionMetaTests` (the two omission guards) with zero exemptions on cross-thread-fork side and the documented stable exemption set on the env-sink side.

Error-path behavior:

- **Cycle in async-spawned global**: spawn an `async fn` whose captured global is a `StashDictionary` containing a cycle. The fork throws a typed error at the cycle site (currently `ValueError` with cycle path; rename to `CycleError` if Open Question is resolved that way). Pre-cycle test fixture: dict X holds a pointer to dict Y holds a pointer to X. Recommended fix: `freeze(X)` (then it shares).
- **Cross-entrypoint cwd**: the CLI run (`stash script.stash`) is **functionally unchanged** even though `env.chdir` no longer mutates `System.Environment.CurrentDirectory`. This works because (Decision Log 2026-06-02-Q2) there is **no general write-through to `System.Environment.CurrentDirectory`** in any mode. Instead, every stdlib sink that previously consulted the real cwd now consults `ctx.WorkingDirectory` (relative-path resolution via `ResolveAgainstCwd`; `process.spawn` seeds `psi.WorkingDirectory` from `ctx.WorkingDirectory`). The one place that historically needs the real cwd to behave correctly is **`process.replace`** (Unix `execvp` inherits the host process's cwd and there is no `WorkingDirectory` argument to `execvp`); this is handled by calling `chdir(ctx.WorkingDirectory)` *just before* `execvp` inside the `process.replace` implementation, scoped to that one bypass. `process.replace` is already embedded-forbidden (`ProcessBuiltIns.cs:102`), so this carve-out only runs in CLI mode. No `MirrorCwdToProcess` flag is needed.

## Phases

Phase ids, deps, files, verify, and `done_when` live in `plan.yaml`. The shape:

- **2A** — Live-hazard fixes (per-VM IC-slot clone + async-child freeze-or-clone globals + `_importStack` per-thread-fork isolation + cross-thread-fork omission guard).
- **2B** — Process-global virtualization (cwd/env on `VMContext` + every stdlib sink routed through it + `ProcessStartInfo` funnel + env-sink omission guard + docs regen + Wave2 throws audit + spec update).

## Open Questions

- **Q1 — Cycle error type.** Should the deep-clone walker throw a typed `CycleError` (new `[StashError]`) or reuse `ValueError`? `CycleError` is clearer at the catch-site and surfaces the cycle path as a typed property, but adds a new error and the `Wave2ThrowsCoverageTests` allow-list / throws metadata churn. **Default for v1:** reuse `ValueError` with the cycle path in the message; a follow-up backlog stub can promote to `CycleError` if it bites.
- **~~Q2 — CLI write-through for cwd.~~** **RESOLVED in Decision Log (2026-06-02): no general write-through.** Every stdlib sink consults `ctx.WorkingDirectory` directly, so neither CLI nor embedded mode needs the real process cwd in sync. The only execvp-style bypass is `process.replace` (CLI-only since it's embedded-forbidden), which calls `chdir(ctx.WorkingDirectory)` immediately before `execvp` — a targeted carve-out, not a general flag. This pins the 2B-3 `done_when` ("`System.Environment.CurrentDirectory` is unchanged after `env.chdir`") as the correct gate in both CLI and embedded modes.
- **Q3 — IC-slot clone representation.** Per-`Chunk` `Dictionary<Chunk, ICSlot[]>` on the VM, vs threading a "VM identity" into IC opcodes and indexing a flat `vm._icSlots[chunkId][slot]`. Dict-on-VM is simpler and the design note's lean default; flat-array is a perf escape hatch if the dict lookup shows in profiles. Implementer owns; the invariant ("no two VMs write the same array") is what the acceptance test guards.
- **Q4 — `EmbeddedMode` propagation.** `EmbeddedMode` is currently a `VMContext.EmbeddedMode` flag mirrored on `VirtualMachine.EmbeddedMode` and set by `StashEngine`. It must also propagate into async-child VMs (today `Async.cs:60,73` already does this). Re-confirm during phase 2A; no change expected.
