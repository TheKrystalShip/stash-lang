# Phase 1A Findings — RuleContext parent visibility and AST grouping

## 1. RuleContext parent visibility

`RuleContext` (at `Stash.Analysis/Rules/RuleContext.cs`) does **NOT** expose any parent-node
information prior to this phase. The only expression-adjacent fields it had were:

- `Expression` — the current node being visited
- `LoopDepth`, `FunctionDepth`, `ElevateDepth` — nesting counters

There is no `Parent`, `ParentExpr`, or `ParentBinaryOperator` field. Approach **(A)** from
spec §3.2 (pure parent check) requires adding one.

### Action taken

Added a nullable `TokenType? ParentBinaryOperator` field to `RuleContext`:

```csharp
/// <summary>
/// The operator type of the immediately enclosing BinaryExpr, or null when the current
/// expression is not a direct child of a BinaryExpr.
/// </summary>
public TokenType? ParentBinaryOperator { get; init; }
```

This field is nullable and defaults to `null`, so all existing rules compile and behave
unchanged.

**The field is not yet populated.** `SemanticValidator.VisitBinaryExpr` must be updated to
pass the operator type into child rule contexts. That update belongs in Phase 1B (see §3 below).

## 2. AST grouping semantics — `(expr)` is NOT transparent

`GroupingExpr` (at `Stash.Core/Parsing/AST/GroupingExpr.cs`) is a **distinct AST node**. It
wraps the inner `Expr` in a `GroupingExpr.Expression` property and is preserved through the
parse. The docstring confirms: "exists as a distinct node (rather than being discarded during
parsing) to preserve the syntactic structure of the original source."

`SemanticValidator.VisitGroupingExpr` visits the inner expression but does NOT call
`DispatchNodeRules` — so rules subscribed to `BinaryExpr` are NOT re-dispatched through a
`GroupingExpr` wrapper.

### Consequence for Phase 1B

`("a" + "b") + name + "c"` produces this AST:

```
BinaryExpr(+)                  ← rule sees this node (outer root)
├── BinaryExpr(+)
│   ├── GroupingExpr           ← NOT a BinaryExpr; chain traversal stops here
│   │   └── BinaryExpr(+)
│   │       ├── LiteralExpr("a")
│   │       └── LiteralExpr("b")
│   └── IdentifierExpr(name)
└── LiteralExpr("c")
```

The `GroupingExpr` acts as a **leaf** from the outer chain's perspective because
`CountStringLiterals` (spec §3.2) only recurses into `BinaryExpr` nodes with `+`. A
`GroupingExpr` is neither a `BinaryExpr` nor a `LiteralExpr`, so it counts as 0 string
literals and contributes no chain extension.

**Result:** `("a" + "b") + name + "c"` has 1 string literal visible to the outer chain
(`"c"` only) plus the `GroupingExpr` leaf (counts 0). Literal count = 1. Does NOT fire at
default threshold 3.

This is the safer, documented behavior per spec §3.2: "If a Group node exists: counted as a
leaf (truncates the chain at that boundary)."

Phase 1B should rely on this semantics as-is — no special `GroupingExpr` handling needed.

## 3. SemanticValidator path discrepancy

The `plan.yaml` entry for phase 1A declares:

```
files:
  - Stash.Analysis/Rules/RuleContext.cs
  - Stash.Analysis/Engines/SemanticValidator.cs
```

The actual file is at **`Stash.Analysis/Visitors/SemanticValidator.cs`** — there is no
`Stash.Analysis/Engines/` directory. The verify-phase scope check would reject any change to
the visitors path for this phase, so `SemanticValidator.cs` was NOT modified in 1A.

**Phase 1B must add `Stash.Analysis/Visitors/SemanticValidator.cs` to its `files` list** (or
the global `scope`) so it can populate `ParentBinaryOperator` in `VisitBinaryExpr`. The
change needed in SemanticValidator is minimal:

```csharp
public object? VisitBinaryExpr(BinaryExpr expr)
{
    DispatchNodeRules(expr);
    // Build child context with parent operator so rules can detect chain-root position.
    // This replaces the generic BuildExprContext call inside DispatchNodeRules for children.
    // -- Actually: DispatchNodeRules calls BuildExprContext internally. The cleanest fix is
    //    to add a ParentBinaryOperator parameter to BuildExprContext and pass it when
    //    dispatching children of a BinaryExpr.
    expr.Left.Accept(this);   // visitor already visits children after dispatch
    expr.Right.Accept(this);
    return null;
}
```

Concretely, the `BuildExprContext` method needs an optional `TokenType?` parent operator
parameter, and `VisitBinaryExpr` must pass `expr.Operator.Type` when visiting children:

```csharp
private RuleContext BuildExprContext(Expr expr, TokenType? parentBinaryOperator = null) => new RuleContext
{
    Expression = expr,
    // ... existing fields ...
    ParentBinaryOperator = parentBinaryOperator,
    // ...
};
```

And in `VisitBinaryExpr`, dispatch children with the parent operator:

```csharp
public object? VisitBinaryExpr(BinaryExpr expr)
{
    DispatchNodeRules(expr);                    // dispatch rules for THIS node (no parent set)
    DispatchChildRules(expr.Left, expr.Operator.Type);
    DispatchChildRules(expr.Right, expr.Operator.Type);
    return null;
}
```

But since the visitor calls `expr.Left.Accept(this)` recursively (not `DispatchNodeRules`
directly on children), the parent operator must be threaded through a field or stack on the
validator. The simplest approach: add a `_parentBinaryOperator` field to `SemanticValidator`,
set it before dispatching children, and reset it after. This is one nullable field, no
structural refactor.

## 4. Approach selection for Phase 1B

Use **Approach (A)**: check `context.ParentBinaryOperator == TokenType.Plus` in
`PreferStringInterpolationRule.Analyze`. If true, the current `BinaryExpr` is a non-root
operand in a `+` chain — skip it. If false (or null), this is the chain root — process it.

This requires Phase 1B to also modify `Stash.Analysis/Visitors/SemanticValidator.cs` to
populate the field.
