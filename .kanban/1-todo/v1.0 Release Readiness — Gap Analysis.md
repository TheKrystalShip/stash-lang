# v1.0 Release Readiness — Gap Analysis

**Status:** Backlog — Analysis
**Created:** 2026-04-19
**Purpose:** Comprehensive audit of what Stash needs before shipping v1.0. Categorized into hard blockers, high-value additions, polish items, and explicitly deferred features.

---

## Executive Summary

Stash is in remarkably good shape. The language core is complete (54 AST nodes, 94 opcodes, 14 data types, full control flow), the tooling is production-quality (LSP with 27 handlers, DAP with 18 handlers, formatter covering all syntax, 68 static analysis rules), and the stdlib is broad (33 namespaces, 492 functions). The package ecosystem (CLI + registry) is functional.

The gaps fall into three categories:

1. **Infrastructure gaps** — no CI/CD, no release automation, no changelog
2. **Stdlib gaps** — missing archive/compression, CSV, logging, and crypto encryption
3. **Quality gaps** — undertested namespaces, runtime error message quality, doc drift

Nothing is architecturally wrong. The work is additive, not corrective.

---

## Tier 0 — Release Blockers

These must be addressed before v1.0. Their absence would make Stash look unfinished to any serious user.

### 0.1 CI/CD Pipeline

**Current state:** No `.github/workflows/` directory exists. Zero automated testing or building.

**Required:**

- Build all projects (`dotnet build`)
- Run all 5,800+ tests (`dotnet test`)
- Cross-platform matrix (Linux, macOS, Windows)
- Native AOT compilation verification (CLI, Check, Format)
- Managed build verification (LSP, DAP)
- Binary size tracking (prevent regression)
- Automated release pipeline (tag → build → publish binaries → GitHub Release)

**Why blocker:** No public project ships without CI. Users need to trust that every commit passes tests. Contributors need automated feedback.

### 0.2 `archive` Namespace — Zip/Tar/Gzip

**Current state:** Zero archive or compression support anywhere in the codebase.

**Required functions:**

```
archive.zip(outputPath, inputPaths, options?)      // Create ZIP archive
archive.unzip(archivePath, outputDir, options?)    // Extract ZIP archive
archive.tar(outputPath, inputPaths, options?)      // Create TAR archive
archive.untar(archivePath, outputDir, options?)    // Extract TAR archive
archive.gzip(inputPath, outputPath?)               // Gzip compress
archive.gunzip(inputPath, outputPath?)             // Gzip decompress
archive.list(archivePath)                          // List contents without extracting
```

**Why blocker:** Every sysadmin script eventually needs to create or extract archives. Deployment, backup, log rotation, artifact management — all require this. Python has `zipfile`/`tarfile`, Go has `archive/zip`/`archive/tar`, PowerShell has `Compress-Archive`. A sysadmin language without archive support is not shippable.

### 0.3 `csv` Namespace — CSV Parse/Stringify

**Current state:** No CSV support. Examples show manual `str.split(line, ",")` which is incorrect (doesn't handle quoting, escaping, newlines in fields).

**Required functions:**

```
csv.parse(text, options?)           // Parse CSV string → array of arrays or array of dicts
csv.stringify(data, options?)       // Array of arrays/dicts → CSV string
csv.parseFile(path, options?)       // Parse CSV file
csv.writeFile(path, data, options?) // Write CSV file
```

Options: `delimiter`, `quote`, `escape`, `header` (bool — first row as keys), `columns` (explicit column names).

**Why blocker:** CSV is the lingua franca of data exchange in system administration. Log files, exports, imports, reports, cloud provider outputs — all use CSV. Manual string splitting is fragile and wrong for real CSV. Every competitor has CSV support built in.

### 0.4 `log` Namespace — Structured Logging

**Current state:** Only `io.print()` and `io.println()`. No log levels, no timestamps, no structured output, no file logging.

**Required functions:**

```
log.debug(message, data?)     // Level: DEBUG
log.info(message, data?)      // Level: INFO
log.warn(message, data?)      // Level: WARN
log.error(message, data?)     // Level: ERROR
log.setLevel(level)           // Filter threshold
log.setFormat(format)         // "text" | "json"
log.setOutput(target)         // "stdout" | "stderr" | filePath
log.withFields(dict)          // Return scoped logger with preset fields
```

**Why blocker:** Sysadmin scripts need observable output. `print()` is not logging — it has no levels, no timestamps, no structured data, no way to filter noise. Python has `logging`, Go has `log/slog`, PowerShell has `Write-Verbose`/`Write-Warning`/`Write-Error`. A sysadmin language without logging is like a car without a dashboard.

### 0.5 Crypto Encryption (AES)

**Current state:** Crypto namespace has hashing (MD5/SHA/HMAC), UUID, and random bytes. No symmetric encryption.

**Required functions:**

```
crypto.encrypt(data, key, options?)     // AES-256-GCM encrypt → { ciphertext, iv, tag }
crypto.decrypt(ciphertext, key, options?)  // AES-256-GCM decrypt → plaintext
crypto.generateKey(bits?)               // Generate random encryption key
```

**Why blocker:** Sysadmin scripts handle secrets — API keys, database passwords, tokens. Without encryption, the only option is plaintext or shelling out to `openssl`. The `secret` type prevents accidental logging but doesn't help with storage. Users will store secrets in plaintext config files because they have no alternative.

### 0.6 CHANGELOG.md

**Current state:** No changelog exists.

**Why blocker:** A v1.0 release without a changelog is unprofessional. Users need to know what changed between versions. Contributors need to understand the project's history.

## Tier 1 — High Value

These would be the first complaints from real users. Not blockers, but strongly expected.

### 1.1 Runtime Error Message Quality

**Current state:** Many runtime errors lack context:

- `"Operands must be numbers."` — doesn't say what types were actually provided
- `"Assignment to constant variable."` — doesn't name the variable
- Type mismatches say "must be X" not "must be X, got Y"
- HTTP/command errors surface raw .NET exception text
- No error codes for runtime errors (only static analysis has SA codes)
- No fix suggestions in runtime errors

**Recommendation:** Audit all `RuntimeError` throws and ensure they include:

1. What was expected
2. What was received
3. Where it happened (source span — already available)
4. Suggestion for fix (where applicable)

### 1.2 Test Coverage Gaps

**Current state:**

- 2 namespaces completely untested: `ini`, `process`
- 6 namespaces with ≤13 tests: `env` (3), `conv` (8), `http` (10), `path` (10), `json` (11), `math` (13)
- `InterpreterTests.cs` is a 1,153-test monolith
- No dedicated test files for interfaces (runtime), closures, or pattern matching

**Recommendation:** Before v1.0, every namespace should have ≥20 dedicated tests. The two untested namespaces are critical — `process` is a core sysadmin namespace.

### 1.3 `io.readPassword(prompt)` — Secret Input

**Current state:** No way to read stdin without echoing. Interactive sysadmin tools need password prompts.

**Required:**

```
let pw = io.readPassword("Enter password: ")  // Returns secret type, no echo
```

### 1.4 TLS for Async TCP

**Current state:** `net.tcpConnectAsync` with `tls: true` throws `"TLS is not yet supported"`. The sync `net.tcpConnect` with TLS works fine.

**Recommendation:** Finish the stub. Users who discover async TCP will expect TLS to work.

### 1.5 `http.head()` — HEAD Requests

**Current state:** Missing. GET/POST/PUT/DELETE/PATCH/request all exist.

**Recommendation:** Trivial addition. Every HTTP client needs HEAD for content-length/existence checks without downloading bodies.

### 1.6 `fs.chown(path, uid, gid)` — File Ownership

**Current state:** Missing. Permissions (rwx) exist but ownership changes don't.

**Recommendation:** Critical for sysadmin on Unix. Windows can be a no-op with a warning.

### 1.7 CONTRIBUTING.md

**Current state:** No contributing guide exists.

**Recommendation:** Standard for open-source projects. Should cover: building, testing, submitting PRs, code style, project structure.

### 1.8 XML Namespace

**Current state:** JSON, YAML, TOML, INI all supported. No XML.

**Required functions:**

```
xml.parse(text)                    // XML string → dict tree
xml.stringify(data, options?)      // Dict tree → XML string
xml.valid(text)                    // Validate well-formedness
xml.query(data, xpath)             // XPath query
```

**Why high-value:** Configuration files (plist, IIS, Maven, Ant), SOAP APIs, RSS/Atom feeds, SVG, HTML parsing — XML is inescapable in system administration.

### 1.9 TAP Framework — `test.only()` and `assert.deepEqual()`

**Current state:**

- No `test.only()` to run a single test during development
- No `assert.deepEqual()` for recursive structural equality (dicts reference-equal only)
- No `assert.closeTo()` for float comparison with tolerance

**Recommendation:** These are table-stakes for a test framework. `test.only()` is the single most-used debugging aid when writing tests. `assert.deepEqual()` is needed for testing any function that returns structured data.

### 1.10 Doc Namespace Count Drift

**Current state:** Multiple docs hardcode "24 namespaces" but actual count is 30+. Already tracked in kanban as a known bug.

**Recommendation:** Fix before v1.0. Incorrect counts in docs undermine trust.

## Tier 2 — Polish

These improve the experience but Stash can ship without them.

### 2.1 Interactive Terminal Helpers

- `term.spinner(text)` — progress spinner for long operations
- `term.progress(current, total)` — progress bar
- `term.select(prompt, options)` — interactive menu
- `term.confirm(prompt)` — yes/no prompt

### 2.2 JSON Query Support

- `json.query(data, path)` — JSONPath or jq-style queries for complex data extraction

### 2.3 Additional DNS Record Types

- `net.resolveNs(domain)` — NS records
- `net.resolveSrv(domain)` — SRV records (service discovery)
- `net.resolveCname(domain)` — CNAME records

### 2.4 Time Helpers

- `time.fromParts(year, month, day, hour?, min?, sec?)` — construct timestamp from components
- `time.measure(fn)` — benchmark wrapper returning `{ result, elapsed }`
- `time.dayOfYear(ts)` — day number within year

### 2.5 Process Spawn Options

- `process.spawn(cmd, { env: {...}, cwd: "..." })` — custom environment and working directory for spawned processes

### 2.6 Additional Crypto

- Password hashing: `crypto.hashPassword(pw)` / `crypto.verifyPassword(pw, hash)` (bcrypt/argon2)
- Key derivation: `crypto.deriveKey(password, salt, options?)` (PBKDF2)
- Digital signatures: `crypto.sign(data, privateKey)` / `crypto.verify(data, signature, publicKey)`

### 2.7 YAML Multi-Document

- `yaml.parseAll(text)` — parse multi-document YAML (common in Kubernetes manifests)
- `yaml.stringifyAll(docs)` — serialize multiple documents

### 2.8 Static Analysis Additions

- Assignment-in-condition detection
- Async function without `await` warning
- Regex pattern validation at analysis time

### 2.9 TAP Extras

- `--output json` / `--output junit` — alternate output formats for CI integration
- Test timeout mechanism (per-test time limits)
- Parallel test execution

### 2.10 VS Code Extension Tasks

- Task provider for `stash` commands (build, test, check, format)
- Auto-generated debug launch configurations

## Tier 3 — Explicitly Deferred (Post-v1.0)

These are features the language spec already declares as future work. Do NOT pursue for v1.0.

| Feature                                          | Reason for Deferral                                                                        |
| ------------------------------------------------ | ------------------------------------------------------------------------------------------ |
| Enum associated values (`Ok(value)`, `Err(msg)`) | Requires significant parser/type system work. Would need generics to be useful.            |
| Generics / type parameters                       | Major language change. Stash is dynamically typed; adding generics is a v2 consideration.  |
| Interface composition (`interface Foo : Bar`)    | Useful but not essential. Structs can implement multiple interfaces.                       |
| Default interface implementations                | Complex interaction with extension methods.                                                |
| Nested struct definitions                        | Syntactic sugar — can define structs separately.                                           |
| Default struct field values                      | Useful but not essential. Constructors serve this role.                                    |
| Class-based inheritance                          | By design — composition over inheritance.                                                  |
| Channels / actors / mutexes                      | Async/await + task.all/race covers most use cases. Structured concurrency is a v2 feature. |
| Generators / yield                               | Would require significant VM changes (coroutines).                                         |
| Comprehensions (list/dict)                       | Nice syntax sugar but `arr.map`/`arr.filter` achieve the same.                             |
| Pattern matching (structural)                    | Would be excellent but is a major language feature.                                        |
| Import aliases (`import { x as y }`)             | Minor convenience.                                                                         |
| Wildcard imports (`import * from`)               | Explicitly avoided — encourages bad practices.                                             |
| Export / visibility modifiers                    | Would require module system redesign.                                                      |
| Raw string literals (`r"..."`)                   | Triple-quoted strings cover most use cases.                                                |
| Type annotations / static checking               | Stash is dynamically typed by design.                                                      |
| SQLite / database connectivity                   | Better as a package (`@stash/sqlite`) than a built-in.                                     |
